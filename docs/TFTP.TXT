TFTP.EXE - TFTP client

Purpose
  Downloads or uploads one file through a TFTP server over ESP-AT UDP. This is
  a minimal RFC 1350 RRQ/WRQ implementation intended to validate the UDP core
  before adding protocol options.

Usage
  TFTP.EXE tftp://host[:port]/path FILE
  TFTP.EXE /PUT FILE tftp://host[:port]/path
  TFTP.EXE /PUT tftp://host[:port]/path FILE
  TFTP.EXE tftp://host[:port]/path PUT FILE

Examples
  TFTP.EXE tftp://192.168.1.36/test.bin TEST.BIN
  TFTP.EXE tftp://192.168.1.36:69/boot/screen.scr SCREEN.SCR
  TFTP.EXE /PUT TEST.BIN tftp://192.168.1.36/upload.bin
  TFTP.EXE /PUT tftp://192.168.1.36/upload.bin TEST.BIN
  TFTP.EXE tftp://192.168.1.36/upload.bin PUT TEST.BIN

Typical sequence
  NETUP.EXE
  TFTP.EXE tftp://192.168.1.36/test.bin TEST.BIN

Testing with macOS
  macOS includes a TFTP server. Put test files into /private/tftpboot and
  start the system daemon from Terminal:

    sudo mkdir -p /private/tftpboot
    sudo cp test.bin /private/tftpboot/test.bin
    sudo chmod 644 /private/tftpboot/test.bin
    sudo launchctl load -w /System/Library/LaunchDaemons/tftp.plist

  Then run from DSS:

    TFTP.EXE tftp://192.168.1.36/test.bin TEST.BIN

  For upload tests, many TFTP servers require the destination file to exist
  and be writable before WRQ is accepted:

    sudo touch /private/tftpboot/upload.bin
    sudo chmod 666 /private/tftpboot/upload.bin

  Then run from DSS:

    TFTP.EXE /PUT TEST.BIN tftp://192.168.1.36/upload.bin

  Stop the macOS server after testing:

    sudo launchctl unload -w /System/Library/LaunchDaemons/tftp.plist

Notes
  The client opens ESP-AT UDP mode 2. This is required because a TFTP server
  normally replies from a transfer-specific UDP port instead of port 69.

  Current limitations:
  - octet mode only;
  - no blksize/timeout/tsize options yet;
  - no directory listing; TFTP is file-only.

Exit status
  0  Transfer completed.
  1  Invalid command line.
  2  Sprinter-WiFi hardware was not found.
  3  Network, ESP or TFTP protocol error.
  5  Local DSS file error.
