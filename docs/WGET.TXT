WGET.EXE - HTTP downloader

Purpose
  Downloads a plain HTTP resource through ESP-AT TCP to a local DSS file.

Usage
  WGET.EXE [http://]host[:port]/path FILE

Examples
  WGET.EXE http://example.com/ INDEX.HTM
  WGET.EXE http://example.com EXAMPLE.HTM
  WGET.EXE example.com EXAMPLE.HTM
  WGET.EXE http://192.168.1.10:8080/file.txt FILE.TXT

Output
  WGET prints the source URL received on the command line and the output file
  name.

  If the URL does not start with http://, WGET treats it as a plain HTTP URL
  and prints a warning that http:// was assumed.

  When the server sends a Content-Length header, WGET prints the expected body
  size before writing the body and prints the downloaded byte count when done.
  If no Content-Length header is present, the size is shown as unknown.

Current limitations
  Plain http:// only. HTTPS is not supported.
  HTTP 2xx responses are downloaded.
  HTTP redirects with an absolute http:// Location header are followed up to
  five hops. HTTPS redirects are reported but cannot be downloaded.
  Chunked transfer and gzip content are not decoded yet.
  Unsupported responses print the received HTTP status line when it is present.

Testing notes
  WGET asks ESP-AT to use passive TCP receive mode. In that mode ESP announces
  available data with +IPD,<len>, and WGET pulls small chunks with
  AT+CIPRECVDATA. This avoids UART overrun while writing data to DSS files.

  If passive receive is not supported by the ESP firmware or emulator, WGET
  falls back to active +IPD mode and prints a warning. Active mode is more
  likely to lose bytes on large downloads.

  WGET depends on TCP receive behavior and is currently the most sensitive tool
  when running under MAME with jesperl. Prefer a local HTTP server known to
  return a simple HTTP/1.0-compatible response while debugging.
  When a Content-Length header is present, WGET verifies that the downloaded
  body size matches it. A mismatch usually means bytes were lost before WGET
  could drain the UART. In MAME/jesperl this can point to missing passive
  receive support, missing RTS/CTS emulation or oversized +IPD bursts. Try a
  lower BAUD value in NET.CFG or use a jesperl version that supports
  AT+CIPRECVMODE and AT+CIPRECVDATA.
  If WGET prints "UART overrun detected", the 16550 UART reported lost input
  bytes. The downloaded file must be treated as corrupted; lower BAUD, run
  NETUP again, and check that NETUP reports UART flow control value 3.
  The TCP core is expected to preserve +IPD payloads that are larger than the
  WGET receive buffer and return them across multiple receive calls.
  WGET keeps its receive buffer below the 0xC000 memory window because DSS file
  writes switch memory pages internally. Buffers crossing that boundary may
  produce files with correct size but corrupted zero-filled blocks.

Exit status
  0  File body downloaded.
  1  Invalid command line or URL.
  2  Sprinter-WiFi hardware was not found.
  3  ESP/TCP/HTTP error.
  5  Local output file error.
